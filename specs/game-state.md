# Game State & Progression

All runtime state lives in a single Zustand store (`src/store/gameStore.ts`). Two `localStorage` keys provide persistence across sessions. React components subscribe to the store via `useGameStore`.

## Decisions

### Single flat Zustand store
All game state — settings, progress, and current-level state — lives in one store. There are no selectors files or separate sub-stores. The store is small enough that a flat shape is simpler than splitting it.

### Two separate localStorage keys
`codino_settings` holds `{ language, apiKey }`. `codino_progress` holds everything needed to resume a game (`initialStory`, `currentLevel`, `completedLevels`, `mapStructure`, `chosenElements`, `stars`). Keeping them separate means `resetProgress()` can delete `codino_progress` without touching `codino_settings`, so the player's API key and language preference survive a reset.

### Settings and progress save only their own fields
Each `saveSettings` call writes exactly `{ language, apiKey }`. Each `saveProgress` call writes exactly the six Progress fields. Spreading the full Zustand state (which includes React callbacks and non-serialisable values) into these keys is a bug — it stores unnecessary data and risks `JSON.stringify` silently dropping fields that happen to shadow serialisable ones.

### `currentLevel` is 0 before any game starts; `selectElement` advances it
`currentLevel` starts at 0 (the "no game in progress" sentinel). Calling `selectElement(element)` appends the element to `chosenElements` and increments `currentLevel` by 1. After selecting the first element, `currentLevel === 1`, meaning "level 1 is in progress". `completeLevel` records stars and adds the level to `completedLevels` but does not change `currentLevel` — the next `selectElement` call advances it again.
> Invariant: `chosenElements.length === currentLevel` whenever `currentLevel > 0`.

### `currentProblem` and `currentCode` are persisted together under a separate key
A third localStorage key, `codino_current_level`, stores `{ problem, code }` while a level is in progress. `setProblem` writes it immediately. `setCode` writes it debounced (2-second delay) so every keystroke does not thrash localStorage. Both fields are stored atomically — they are always written together to guarantee the stored code always matches the stored problem.

The key is cleared on `completeLevel`, `selectElement`, and `resetProgress`. On the next app load, if the key exists, `currentProblem` and `currentCode` are restored from it, so a child who accidentally refreshes mid-level returns to exactly the state they left.

`codino_current_level` is deliberately separate from `codino_progress`: progress represents canonical game state that can reconstruct a session; the current-level key is ephemeral and only meaningful while that specific level is open. Clearing progress does not leave behind orphaned current-level state.

> Alternatives considered: persisting `currentCode` alone (without `currentProblem`) was rejected because on refresh the problem would be regenerated by Claude — possibly with different wording — making the restored code incoherent with the new problem. Adding `currentProblem` to `codino_progress` was rejected because progress should only hold state needed to resume the game at the map level, not mid-editor state.

### `loadSettings` and `loadProgress` return safe defaults on parse failure
Both functions wrap `JSON.parse` in a `try/catch`. A corrupted or unreadable localStorage entry returns the same defaults as a missing one (`{ language: 'en', apiKey: null }` / `{}`), so the app never crashes on startup due to bad persisted data.

### `resetProgress` preserves settings
`clearProgress()` removes only `codino_progress`. The store resets all progress fields to their initial values but leaves `language` and `apiKey` unchanged, so the player does not need to re-enter their API key after clearing a game.

## Invariants

INV-01: `chosenElements.length === currentLevel` whenever `currentLevel > 0`.

INV-02: For every `n` in `completedLevels`, `stars[n]` is a number in `[1, 3]`.

INV-03: `currentLevel` is in `[0, 10]`. The game ends when level 10 is completed.

INV-04: `saveSettings` writes exactly `{ language, apiKey }` — no other fields.

INV-05: `saveProgress` writes exactly `{ initialStory, currentLevel, completedLevels, mapStructure, chosenElements, stars }` — no other fields.

INV-06: `loadSettings()` and `loadProgress()` never throw. On any parse failure they return `{ language: 'en', apiKey: null }` and `{}` respectively.

INV-07: `resetProgress()` leaves `language` and `apiKey` in the store and in `localStorage` unchanged.

INV-08: `currentProblem` and `currentCode` are persisted together under `codino_current_level`. They are never written to `codino_progress` or `codino_settings`.

INV-09: The `codino_current_level` key is cleared atomically with `completeLevel`, `selectElement`, and `resetProgress`. After any of these calls the key does not exist in localStorage.

INV-10: The debounce timer for `setCode` is always cancelled before `clearCurrentLevel` is called, so a pending save can never overwrite a cleared key after a level ends.
