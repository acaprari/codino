Code Editor Experience

How should the problem-solving interface look?

📖 THE CHALLENGE
The brave knight found a magical sword in the castle! The sword can defeat 3 dragons at once.

Help the knight: If the sword defeats 3 dragons, how many dragons are left from 10?
Expected output: 7
1 draghi = 10
2 sconfitti = 3
3 rimasti = draghi - sconfitti
4 STAMPA rimasti

A. Side-by-Side Split

Story/problem on left, code editor on right. Both visible at once. Good for referencing problem while coding.

  • See problem always
  • Natural flow
  • Feels professional
  • Less space for each
  • Might feel cramped

B. Top-Bottom Stack

Problem on top, editor below. More horizontal space for code. Classic "read then code" flow.

  • More code width
  • Read-then-code flow
  • Works on tablets
  • Can't see problem while coding below

C. Floating Problem Card

Full-screen editor with problem in a collapsible card (top-right). Max coding space, but problem can be toggled away.

  • Maximum editor space
  • Minimalist, focused
  • Problem dismissible when memorized
  • More complex UI
  • Card might obscure code

Editor Features

Regardless of layout, these features should be included:

✅ Autocomplete
Type "RIP" → suggests "RIPETI". Visual dropdown with keywords in both languages.
🎨 Syntax Highlighting
Keywords (RIPETI, SE) in blue, numbers in yellow, variables in pink. Makes code readable.
😊 Friendly Errors
"Hai scritto RIPETTI — intendevi RIPETI? 🤔" Appears inline or in a panel below code.
✨ Visual Execution
Line-by-line animation showing which line is running. Variables update visually in sidebar/overlay.
Question: Should execution be animated (showing each line highlighting as it runs) or just show final output? Animation helps understanding but takes more time.
